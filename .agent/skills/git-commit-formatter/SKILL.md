# Git Commit Formatter

## Purpose
To ensure a consistent, clean, and semantic git history by strictly adhering to the **Conventional Commits** specification.

## Rules
When asked to generate a commit message or perform a git commit, you MUST follow this format:

```
<type>(<scope>): <subject>

<body>

<footer>
```

### 1. Types
Use strictly one of the following types:
- **feat**: A new feature
- **fix**: A bug fix
- **docs**: Documentation only changes
- **style**: Changes that do not affect the meaning of the code (white-space, formatting, missing semi-colons, etc)
- **refactor**: A code change that neither fixes a bug nor adds a feature
- **perf**: A code change that improves performance
- **test**: Adding missing tests or correcting existing tests
- **chore**: Changes to the build process or auxiliary tools and libraries such as documentation generation

### 2. Scope (Optional but Recommended)
A strictly enclosed scope in parentheses describing the section of the codebase affected (e.g., `(hero)`, `(header)`, `(api)`).

### 3. Subject
- Imperative tone ("add" not "added", "change" not "changes").
- No capital letter at start.
- No period (.) at the end.
- Concise (under 50 chars if possible).

### 4. Body (Optional)
- imperative tone.
- Explain "what" and "why", not "how".

### Examples

**Good:**
`feat(hero): implement 3d slot machine animation`
`fix(pagination): resolve index overflow in slider`
`chore: update implementation plan`

**Bad:**
`Added a new feature` (No type)
`Fixed the bug` (Passive voice)
`feat(Header) - Fixed styling` (Incorrect casing/punctuation)
